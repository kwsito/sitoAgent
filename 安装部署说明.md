# 安装部署说明（开源版 / AppAgent_mobile）

## 1. 运行形态

- **开发机**：Windows（推荐 PowerShell 7）
- **构建环境**：WSL（Ubuntu）+ buildozer + python-for-android
- **运行环境**：Android 真机/模拟器（开启 USB 调试）
- **设备交互**：ADB（截图、XML、输入事件）

## 2. 必备条件

### 2.1 Windows 侧

- 安装 **ADB / platform-tools**，并确保 `adb` 在 PATH 中
- 安装 **WSL**（Ubuntu），可执行 `wsl.exe -e bash -lc "echo ok"`

### 2.1.1 WSL 安装方法

推荐以**管理员身份**打开 PowerShell，按下面顺序执行：

```powershell
wsl --install
```

如果希望指定 Ubuntu：

```powershell
wsl --install -d Ubuntu-22.04 --web-download
```

安装完成后需要**重启电脑**。重启后继续执行：

```powershell
wsl --set-default-version 2
wsl -l -v
```

如果安装成功，你会看到类似下面的输出：

```text
  NAME           STATE      VERSION
* Ubuntu-22.04   Stopped    2
```

第一次进入 Ubuntu：

```powershell
wsl
```

系统会提示你创建 Linux 用户名和密码。

### 2.1.2 WSL 常见问题与解决方案

**问题 1：`wsl --install` 进度一直在 0%**

通常不是卡死，而是系统组件还没启用完成。先手动启用：

```powershell
dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
```

然后执行一次完整重启：

```powershell
shutdown /r /t 0 /f
```

重启后再执行：

```powershell
wsl --update --web-download
wsl --set-default-version 2
wsl --install -d Ubuntu-22.04 --web-download
```

**问题 2：一直提示“请求的操作成功。直到重新启动系统前更改将不会生效。”**

这说明 WSL 相关组件已经安装，但系统仍然认为有待重启的虚拟化配置没有真正生效。先检查状态：

```powershell
dism.exe /online /get-featureinfo /featurename:Microsoft-Windows-Subsystem-Linux
dism.exe /online /get-featureinfo /featurename:VirtualMachinePlatform
```

如果二者都显示“已启用”，继续检查 hypervisor 启动项：

```powershell
bcdedit /enum | findstr -i hypervisorlaunchtype
```

如果没有任何输出，执行：

```powershell
bcdedit /set hypervisorlaunchtype auto
shutdown /r /t 0 /f
```

重启后再次执行：

```powershell
wsl --status
wsl --set-default-version 2
wsl --install -d Ubuntu-22.04 --web-download
```

**问题 3：报错 `HCS_E_HYPERV_NOT_INSTALLED` / “当前计算机配置不支持 WSL2”**

这表示 Ubuntu 已经开始下载/安装，但在创建 WSL2 虚拟机时失败。通常原因有两个：

- BIOS / UEFI 没有开启虚拟化
- Windows hypervisor 没有正常启动

先确认：

```powershell
bcdedit /set hypervisorlaunchtype auto
dism.exe /online /enable-feature /featurename:HypervisorPlatform /all /norestart
shutdown /r /t 0 /f
```

然后检查：

```powershell
bcdedit /enum | findstr -i hypervisorlaunchtype
systeminfo
```

如果 `systeminfo` 里显示虚拟化不可用，就需要进 BIOS 开启：

- Intel：`Intel Virtualization Technology` / `VT-x`
- AMD：`SVM Mode` / `AMD-V`

**问题 4：`Microsoft-Hyper-V-All` 提示未知功能名称**

这通常出现在 Windows 家庭版，属于正常现象，不一定影响 WSL2。WSL2 的关键依赖仍然是：

- `Microsoft-Windows-Subsystem-Linux`
- `VirtualMachinePlatform`
- `HypervisorPlatform`
- `hypervisorlaunchtype auto`

所以如果 `Microsoft-Hyper-V-All` 不存在，可以跳过，不必强求开启完整 Hyper-V。

**问题 5：如何快速确认 BIOS 虚拟化是否已启用**

可以打开任务管理器：

- 进入“性能”
- 选择 CPU
- 查看右下角“虚拟化”

如果显示：

- `已启用`：说明 BIOS 基本已开启虚拟化
- `已禁用`：需要进 BIOS 打开虚拟化

### 2.1.4 WSL 内安装 Buildozer 与构建依赖

如果在执行构建脚本时看到类似报错：

- `stdbuf: failed to run command 'buildozer': No such file or directory`

说明 WSL（Ubuntu）已经能运行，但里面还没有安装 `buildozer` 及 Android 打包依赖。可以直接进入 WSL 后执行下面这一整段：

```bash
sudo apt update && sudo apt install -y \
  python3 python3-pip python3-venv zip unzip git openjdk-17-jdk \
  autoconf automake m4 libtool libtool-bin pkg-config gettext zlib1g-dev libncurses-dev \
  cmake libffi-dev libssl-dev build-essential ccache \
  libltdl-dev libsqlite3-dev libjpeg-dev libfreetype-dev && \
  python3 -m pip install --user --break-system-packages --upgrade pip setuptools wheel && \
  python3 -m pip install --user --break-system-packages buildozer cython && \
  echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc && \
  export PATH="$HOME/.local/bin:$PATH" && \
  buildozer --version
```

执行方式：

```powershell
wsl
```

进入 Ubuntu 后把上面那整段命令粘贴进去执行即可。

说明：

- Ubuntu 24.04（noble）下已经没有 `libtinfo5`
- `libncurses5-dev` / `libncursesw5-dev` 可直接改用 `libncurses-dev`
- `libfreetype6-dev` 可直接改用 `libfreetype-dev`
- Ubuntu 24.04 / Python 3.12 默认启用了 PEP 668，所以 `pip install --user` 需要追加 `--break-system-packages`
- `autoreconf` / `libffi` 构建阶段还会依赖 `automake`、`m4`、`libtool-bin`、`gettext`
- 如果构建日志提示“检测到宿主 gcc=13，但未安装 gcc-12/gcc-11/clang”，请先安装下面任意一组编译器：

```bash
sudo apt update && sudo apt install -y gcc-12 g++-12
```

或：

```bash
sudo apt update && sudo apt install -y clang
```

安装完成后，可执行下面命令确认：

```bash
gcc-12 --version
clang --version
```

然后重新执行：

```powershell
.\one_click_build_install.ps1
```

如果构建日志里出现下面这类错误：

- `possibly undefined macro: AM_INIT_AUTOMAKE`
- `possibly undefined macro: AM_PROG_AS`
- `possibly undefined macro: AC_PROG_LIBTOOL`
- `autoreconf: error: /usr/bin/autoconf failed with exit status: 1`

通常说明 WSL 里缺少完整 autotools 组件。请执行：

```bash
sudo apt update && sudo apt install -y autoconf automake m4 libtool libtool-bin gettext
which autoreconf aclocal automake libtoolize
autoreconf --version
aclocal --version
automake --version
libtoolize --version
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/build/other_builds/libffi
```

然后重新执行构建脚本。

如果你已经安装过这些包，但错误仍然完全一样，建议直接重装相关组件并清理整段构建缓存：

```bash
sudo apt update && sudo apt install --reinstall -y autoconf automake m4 libtool libtool-bin gettext
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a
```

然后重新执行构建脚本。

如果构建日志长时间卡在下面这类信息：

- `Downloading openssl`
- `openssl-1.1.1w.tar.gz`
- 下载进度长时间不动，或最终 404 / 超时

通常是上游 OpenSSL 旧版本源码地址已迁移，导致 python-for-android 默认下载源不稳定。当前仓库的构建脚本已经增加了 openssl 预下载逻辑，会优先尝试：

- `https://www.openssl.org/source/old/1.1.1/openssl-1.1.1w.tar.gz`

如果你已经更新到最新仓库脚本，直接重新执行：

```powershell
.\one_click_build_install.ps1
```

如果仍想手动处理，也可以在 WSL 中先删除 openssl 包缓存后重试：

```bash
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages/openssl
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/build/other_builds/openssl
```

然后再次执行构建脚本。

如果构建日志继续卡在下面这类信息：

- `Downloading sdl2_image`
- `SDL2_image-2.8.0.tar.gz`
- 下载进度长时间不动，或 GitHub 下载极慢 / 超时
- 或后续报错：`external/libwebp/Android.mk: No such file or directory`

处理方式和 openssl 类似。当前仓库构建脚本已增加 `sdl2_image` 预下载逻辑，会优先尝试：

- `https://github.com/libsdl-org/SDL_image/releases/download/release-2.8.0/SDL2_image-2.8.0.tar.gz`

如果你已经同步到最新脚本，直接重新执行：

```powershell
.\one_click_build_install.ps1
```

如果想手动清缓存后再试，可执行：

```bash
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages/sdl2_image
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/build/other_builds/sdl2_image
rm -rf ~/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/build/bootstrap_builds/sdl2/jni/SDL2_image
```

然后再次执行构建脚本。

### 2.1.5 导出 / 复用已下载的构建缓存

如果这台机器已经把大部分依赖包成功下载过，强烈建议把缓存导出保存。这样在新环境里可以直接恢复，避免再次经历：

- `Downloading openssl`
- `Downloading sdl2_image`
- `Downloading sdl2_mixer`
- `Downloading cryptography`
- 以及各种 GitHub / 上游下载超时、404、断流问题

#### 推荐备份的目录

优先备份下面两类：

```bash
$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages
$HOME/.buildozer/android/packages
```

如果你还想把 SDK / NDK 一起复用，也可以额外备份：

```bash
$HOME/.buildozer/android/platform/android-sdk
$HOME/.buildozer/android/platform/android-ndk-r25b
```

#### 导出轻量缓存包

在 WSL 中执行（示例导出到 Windows 的 D 盘）：

```bash
mkdir -p /mnt/d/WSL/cache_backup
tar -cpf /mnt/d/WSL/cache_backup/buildozer-packages-backup.tar \
  "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages" \
  "$HOME/.buildozer/android/packages"
```

#### 导出完整缓存包（包含 SDK / NDK）

```bash
mkdir -p /mnt/d/WSL/cache_backup
tar -cpf /mnt/d/WSL/cache_backup/buildozer-full-backup.tar \
  "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages" \
  "$HOME/.buildozer/android/packages" \
  "$HOME/.buildozer/android/platform/android-sdk" \
  "$HOME/.buildozer/android/platform/android-ndk-r25b"
```

#### 在新环境中恢复缓存

推荐先把 `buildozer-packages-backup.tar` 放到新机器的 Windows 盘，例如：

```text
D:\WSL\cache_backup\buildozer-packages-backup.tar
```

在 WSL 里它对应：

```bash
/mnt/d/WSL/cache_backup/buildozer-packages-backup.tar
```

**方式 A：新环境 Linux 用户名与旧环境一致**

如果新机器上的 Linux 用户名和打包机器一致，可以直接恢复到根目录：

```bash
mkdir -p "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a"
mkdir -p "$HOME/.buildozer/android"
tar -xpf /mnt/d/WSL/cache_backup/buildozer-packages-backup.tar -C /
```

如果你导出的是完整缓存包，则改为：

```bash
tar -xpf /mnt/d/WSL/cache_backup/buildozer-full-backup.tar -C /
```

**方式 B：新环境 Linux 用户名不同（更推荐）**

如果新机器用户名不同，最稳的方式是**按新用户名直接改写 tar 包内路径再解压**。假设旧机器用户名是 `wangxiao`，新机器当前用户就是 `$USER`，执行：

```bash
mkdir -p "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a"
mkdir -p "$HOME/.buildozer/android"
tar -xpf /mnt/d/WSL/cache_backup/buildozer-packages-backup.tar \
  --transform="s#^home/wangxiao/#home/$USER/#" \
  -C /
```

如果你导出的是完整缓存包，则改为：

```bash
tar -xpf /mnt/d/WSL/cache_backup/buildozer-full-backup.tar \
  --transform="s#^home/wangxiao/#home/$USER/#" \
  -C /
```

如果你不确定旧用户名，或者不想直接改写路径，也可以先解压到临时目录，再同步到当前用户目录：

```bash
mkdir -p "$HOME/restore_tmp"
tar -xpf /mnt/d/WSL/cache_backup/buildozer-packages-backup.tar -C "$HOME/restore_tmp"
mkdir -p "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a"
mkdir -p "$HOME/.buildozer/android"
rsync -a "$HOME/restore_tmp/home/"*/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages/ \
      "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages/"
rsync -a "$HOME/restore_tmp/home/"*/.buildozer/android/packages/ \
      "$HOME/.buildozer/android/packages/"
```

如果你导出的是完整缓存包，还可以继续恢复 SDK / NDK：

```bash
mkdir -p "$HOME/.buildozer/android/platform"
rsync -a "$HOME/restore_tmp/home/"*/.buildozer/android/platform/android-sdk/ \
      "$HOME/.buildozer/android/platform/android-sdk/"
rsync -a "$HOME/restore_tmp/home/"*/.buildozer/android/platform/android-ndk-r25b/ \
      "$HOME/.buildozer/android/platform/android-ndk-r25b/"
```

#### 恢复后建议检查

```bash
ls -l "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages"
ls -l "$HOME/.buildozer/android/packages"
find "$HOME/mobile_build_workspace/.buildozer/android/platform/build-arm64-v8a/packages" -maxdepth 2 -name ".mark-*" | head
which buildozer
buildozer --version
```

如果你恢复了完整缓存，也建议检查：

```bash
ls -l "$HOME/.buildozer/android/platform/android-sdk"
ls -l "$HOME/.buildozer/android/platform/android-ndk-r25b"
```

恢复完成后，就可以重新执行项目构建脚本：

```powershell
.\one_click_build_install.ps1
```

#### 注意事项

- 文档中的路径建议优先使用 `$HOME`，不要写死成某个用户名目录
- 如果新环境用户名不同，直接恢复到 `/` 前先确认 tar 包内路径是否与你当前用户匹配
- 这些缓存不建议提交到 git，但非常适合本地备份或团队内部共享
- 最佳实践是：**首次成功构建后立刻备份缓存**
- 如果恢复后 `.\one_click_build_install.ps1` 仍然重新下载，通常是因为：
  - tar 包内原始路径和当前用户路径不一致，缓存没有恢复到正确目录
  - `packages` 内的隐藏 `.mark-*` 文件没有被正确恢复
  - 只恢复了轻量缓存，没有恢复 SDK / NDK，因此工具链仍会下载其他内容

## 3. 配置（非常重要）

### 3.1 配置多模态模型 Key

配置文件：[config.yaml](file:///g:/AppAgent/AppAgent_mobile_bak/config.yaml)

- `ARK_API_KEY`
- 不配置 key 时，执行器会提示未配置并跳过/失败（取决于运行路径）

### 3.2 ADB 目录配置

`config.yaml` 中：

- `ANDROID_SCREENSHOT_DIR`：设备侧保存截图的目录（默认 `/sdcard/AppAgent/screenshots`）
- `ANDROID_XML_DIR`：设备侧保存 XML 的目录（默认 `/sdcard/AppAgent/xml`）

一般不需要改；如果你改了，请确保目录存在（脚本会尝试 `mkdir -p`，但某些 ROM 权限策略不同）。

## 4. 构建与安装（推荐方式）

项目根目录：`G:\AppAgent\AppAgent_mobile_bak`

### 4.1 一键构建 + 安装（推荐）

```powershell
.\one_click_build_install.ps1
```

- 会在 WSL 内执行 [build_apk.sh](file:///g:/AppAgent/AppAgent_mobile_bak/build_apk.sh) 完整构建
- 构建完成后 APK 会复制回 Windows 项目根目录
- 然后自动执行 `adb install -r` 安装到目标设备

如果你电脑连接了多台设备，请手动指定：

```powershell
.\one_click_build_install.ps1 -Serial <adb_device_serial>
```

如果你想强制先卸载再安装（更干净）：

```powershell
.\one_click_build_install.ps1 -ForceUninstall
```

### 4.2 快速构建（开发迭代）

```powershell
.\quick_build.ps1 -Install
```

遇到构建缓存/奇怪错误时：

```powershell
.\quick_build.ps1 -Clean -Install
```

如果遇到类似下面的报错：
- `ModuleNotFoundError: No module named 'setuptools'`（通常发生在 Pillow 编译阶段）

解决方法：
- 确保 [buildozer.spec](file:///g:/AppAgent/AppAgent_mobile_bak/buildozer.spec) 的 `requirements` 包含 `setuptools,wheel`
- 然后重新执行一次完整构建：

```powershell
.\one_click_build_install.ps1
```

## 7. 文件精简建议（部署包/仓库）

以下内容建议不要长期保留在仓库中（运行时会自动生成）：

- `*.apk`、`bin/`、`downloaded_apk/`
- `logs/`、`tasks/`、`*.log`、调试截图（例如 `*output*.png`）

仓库已提供 `.gitignore`，默认会忽略上述产物文件。
