# 技术路线报告（AppAgent_mobile）

## 1. 项目目标

本项目将 AppAgent 的“手机自动化执行”能力封装进 Android 端 Kivy 应用，通过：

- 拉取/接收任务（订单、设备ID、任务描述）
- 通过 ADB 采集手机当前界面（截图 + UI XML）
- 对界面进行可交互元素标注（生成 labeled 图或 grid 图）
- 调用多模态大模型生成下一步动作（tap/text/long_press/swipe/grid 等）
- 将动作回放到设备并循环，直到任务完成或达到终止条件

## 2. 端到端数据流

一次任务的核心循环（按轮次 round）：

1. **创建任务目录**（Android 私有目录）
   - 目录形如：`.../files/tasks/<device>_task_<app>_<timestamp>/`
2. **采集屏幕**
   - `screencap -p` 生成 PNG
   - `uiautomator dump` 生成 XML
3. **解析 XML -> 元素列表**
   - 从 XML 中提取 clickable/focusable 节点
   - 做去重/距离过滤，得到 `elem_list`
4. **生成输入图片（两种模式二选一）**
   - **labeled 模式**：对 PNG 叠加数字标签，输出 `*_labeled.png`
   - **grid 模式**：对 PNG 叠加网格编号，输出 `*_grid.png`
5. **构造 Prompt + 图片 -> 调用多模态模型**
   - `scripts/model.py` 将图片编码（base64 或 file://）并请求模型
6. **解析模型输出 -> 动作执行**
   - 解析为 `tap(n)` / `text(xxx)` / `swipe(n, dir, dist)` / `grid` 等
   - 通过 `scripts/and_controller.py` 执行 ADB 输入事件
7. **日志与状态回传**
   - 记录每轮 response、停止原因、任务完成状态等

## 5. 关键容错策略

### 5.1 labeled 图生成失败不再导致任务崩溃

旧链路问题：`*_labeled.png` 未生成 -> 模型侧读取图片路径失败 -> 任务异常退出。

现在的处理：

- `draw_bbox_multi()` 返回值与文件存在性校验
- 如果 labeled 生成失败：
  - 本轮跳过
  - 下轮切换到 grid 模式继续

### 5.2 图像标注库缺失的降级

在 Android APK 中 `opencv-python (cv2)` 往往不可用，表现为 `No module named 'cv2'`。

现在：

- 优先尝试 OpenCV（如果可用）
- 不可用则自动 fallback 到 PIL，仍可生成 labeled 图

### 5.3 截图/XML 获取的重试与诊断

截图或 XML 获取失败时：

- 支持多次重试（间隔可配置）
- 输出诊断信息（ADB 命令、目录权限、文件存在性等）
- 超过阈值后以 `capture_failed` 终止任务，避免死循环
